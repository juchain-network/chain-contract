# PoSA 本地测试系统 - 测试路径规划

## 0.背景信息

这个合约系统需要结合 Go 的共识逻辑，以驱动合约的状态变化。Go 共识交互逻辑与合约的具体调用关系如下：

### 1. 验证者集合管理
- **定期检查验证者**：共识层每隔固定周期调用 `Validators.getActiveValidators()` 获取当前活跃验证者列表
- **更新活跃验证者**：纪元切换时调用`Validators.getTopValidators()`获取按权重排序的顶级验证者列表，并调用 `Validators.updateActiveValidatorSet()` 更新活跃验证者集合
- **检查验证者状态**：出块前调用 `Validators.isActiveValidator(address)` 验证节点是否为活跃验证者

### 2. 验证者权重计算
- **获取验证者信息**：共识层调用 `Staking.getValidatorInfo(address)` 获取验证者质押量和表现数据
- **计算验证者权重**：根据质押量、验证历史等计算验证者权重
- **获取顶级验证者**：调用 `Staking.getTopValidators(uint256)` 获取按权重排序的顶级验证者

### 3. 出块权分配
- **按权重分配出块顺序**：根据计算出的验证者权重分配出块顺序
- **出块前验证**：调用 `Validators.isValidatorExist(address)` 验证出块节点是否为有效验证者

### 4. 区块奖励分发
- **出块后调用**：区块产生后，共识层调用 `Validators.distributeBlockReward{value: feeAmount}()` 分发交易手续费
- **奖励分配**：调用 `Staking.distributeRewards{value: blockReward}()` 分发出块奖励

### 5. 纪元管理
- **纪元结束检查**：每个区块产生时检查是否达到纪元结束条件
- **纪元切换触发**：纪元结束时，调用 `Validators.updateActiveValidatorSet()` 更新验证者集合

### 6. 验证者惩罚
- **异常检测**：共识层检测到验证者异常行为（如错过区块、双重签名）
- **惩罚调用**：调用 `Punish.punish(address)` 对异常验证者进行惩罚，内部惩罚逻辑会根据情况进行罚没交易费、踢出顶层验证者、监禁质押状态等
- **检查惩罚状态**：调用 `Punish.getPunishRecord(address)` 获取验证者惩罚记录

### 7. 提案处理
- **检查提案有效性**：调用 `Proposal.isProposalValidForStaking(uint256)` 检查提案是否在有效期内
- **获取提案信息**：调用 `Proposal.getProposal(uint256)` 获取提案详情
- **执行通过的提案**：对投票通过的提案，如提案新验证者，则会记录提案通过，若配置修改提案，则会更新配置项为新值

### 8. 共识驱动的状态变化
- **合约初始化**：系统启动时调用各合约的 `initialize()` 方法进行初始化
- **验证者注册处理**：当提案通过后，新验证者第一次需要调用`Staking.registerValidator`注册自己，
- **验证者信息管理**: 可外部调用 `Validators.createOrEditValidator()` 更新验证者信息
- **验证者激活**：当验证者被监禁后，如果再次通过提案，则需要自行调用`Staking.unjailValidator`再次激活自己(不需要再次注册)
- **验证者退出处理**：调用 `Staking.resignValidator()` 退出自己的验证者身份，然后等到下一个 epoch 再次调用`Staking.exitValidator`申请退出质押，然后待定固定的周期后即可调用`Staking.withdrawUnbonded`提取解绑的质押代币

### 9. 验证者退出机制分析

当前系统没有实现紧急停止机制，但验证者退出机制设计合理，主要包括以下几种方式：

1. **主动退出机制**
   - 验证者可以调用 `Staking.resignValidator()` 主动辞去验证者角色
   - 辞去角色后，验证者会被标记为jailed状态，并在下次纪元切换时从活跃验证者集合中移除
   - 完全退出需要调用 `Staking.exitValidator()`，但必须先调用 `resignValidator()` 并等待一个纪元周期，确保系统稳定性

2. **提案踢除机制**
   - 其他验证者可以通过提案系统将不良验证者踢出去
   - 提案通过后，会调用 `Validators.tryRemoveValidator(address)` 移除验证者
   - 该方法会先监禁验证者，然后将其从活跃验证者集合中移除

3. **监禁机制**
   - 对违规验证者（如错过区块、双重签名），会调用 `Punish.punish()` 进行惩罚
   - 严重违规的验证者会被调用 `Staking.jailValidator()` 监禁
   - 监禁期间，验证者无法参与出块和投票

## 2.5 测试背景信息

### 2.5.1 验证者注册激活机制

1. **初始化验证者**
   - 通过`initialize`或`initializeWithValidators`方法传递的验证者列表会自动注册并激活
   - 测试环境需要提前给Staking合约充值，作为这些验证者的质押
   - 这些验证者无需调用`registerValidator`方法

2. **提案加入的验证者**
   - 通过提案系统加入的验证者需要调用`registerValidator`方法注册身份
   - 调用`registerValidator`时需要发送质押代币
   - `registerValidator`只需要调用一次

3. **验证者再次成为验证者**
   - 如果验证者退出后未退出质押，再次通过提案成为验证者时，只需调用`unjailValidator`解禁即可
   - 如果验证者已经退出质押，再次通过提案成为验证者后，需要先调用`addValidatorStake`添加质押，然后才能调用`unjailValidator`解禁

**路径4：注册并添加质押后退出**

- 调用`exitValidator`
- 调用`withdrawUnbonded`提取质押

**路径5：注册、添加质押、减少质押后退出**
