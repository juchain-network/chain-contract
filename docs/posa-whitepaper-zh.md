# JPoSA 共识白皮书

## 摘要

JPoSA 是 JuChain 的 Proof of Stake Authority 设计。它把基于权益的验证者竞争和显式链上治理授权结合在一起。

这个系统既不是纯粹的 PoS 竞价模型，也不是纯粹的 PoA 白名单模型：

- 质押决定排序和奖励权重
- 治理决定谁有资格进入或重新进入验证者池
- Congress 共识只在 epoch 边界切换验证者集合

本文总结当前 PoSA 系统合约和 Congress 共识引擎的实际行为。

## 核心亮点

### 1. 混合共识

JPoSA 同时吸收了 PoS 和 PoA 的主要优点：

- 通过自质押加委托实现 PoS 风格的竞争
- 通过治理提案实现 PoA 风格的准入控制
- 通过 epoch 切换维持运行稳定性

### 2. 治理控制的验证者准入

创世之后，成为验证者需要两个步骤：

1. 先通过治理授权
2. 再完成质押注册

这意味着仅有质押并不足以进入验证者池。

### 3. 双通道奖励模型

JPoSA 明确区分：

- 交易费收益
- Coinbase 区块奖励

交易费路径由 `Validators` 处理，Coinbase 奖励由 Congress 计算并通过 `Staking` 分发。

### 4. 动态但可预测的验证者集合

验证者集合是动态变化的，但不是每次状态变化都会立刻影响共识。

- 候选排序会随着质押变化而变化
- 实际出块集合只在 epoch 边界切换
- 被 jailed 的验证者会立刻失去关键权限，即使 epoch 缓存还没轮换

### 5. 可治理的经济参数

像区块奖励输入、惩罚阈值、解绑期、验证者上限、佣金上限和 slash 金额等核心参数，都可以通过治理调整，而不需要重部署系统合约。

## 治理机制

JPoSA 的治理由四个协同子系统组成。

### 提案系统

提案系统是网络决策的入口。

它支持两类提案：

1. 验证者提案
   - 新增或移除验证者授权
2. 配置提案
   - 更新可治理协议参数

投票规则：

- 只有活跃且未 jailed 的验证者可以投票
- 门槛是投票验证者数量的一半以上
- 一旦过半，提案立即生效
- 提案结果不会等到过期后再统一结算

`proposalLastingPeriod` 仍然重要，但它承担两个窗口作用：

- 投票有效窗口
- 新验证者在提案通过后完成注册的窗口

### 验证者系统

验证者系统维护网络使用的验证者视图。

它负责：

- epoch 生效的活跃集合
- 用于排序的候选缓存
- 验证者费地址和元数据
- 交易费收益余额

因此，`Validators` 是治理、质押状态和共识引擎之间的桥梁。

### 质押系统

质押系统管理所有权益相关的经济状态：

- 验证者自质押
- 委托
- 撤销委托和解绑
- 验证者与委托人的奖励记账
- jail 状态

验证者也在这里完成：

- 通过治理后的注册
- 主动辞任
- 退出
- unjail

### 惩罚系统

惩罚系统负责执行活性和安全规则。

它负责：

- 跟踪漏块计数器
- 在较低阈值移除交易费收益资格
- 在较高阈值执行 jail 和移除
- 处理双签证据
- 在 epoch 边界衰减漏块计数

这四个系统共同提供了治理、经济激励和运行约束，而不是把所有逻辑挤在单个合约里。

## 经济模型

JPoSA 的激励模型围绕权益参与、验证者表现和可治理参数展开。

### 奖励来源

系统存在两种不同的奖励来源：

1. 交易费收益
   - 通过 `Validators` 路径入账
   - 若出块者未被 jailed，则计入该验证者
2. Coinbase 区块奖励
   - 由 Congress 计算
   - 通过 `Staking` 分配

这一区分在运行上很重要，因为两类余额的记账和提现路径不同。

### Coinbase 奖励公式

治理参数 `blockReward` 只是基础输入，并不是每个区块的固定最终收益。

Congress 实际按下式计算生产者奖励：

```text
fixedPart    = blockReward * baseRewardRatio / 10000
weightedPool = blockReward * validatorCount * (10000 - baseRewardRatio) / 10000
weightedPart = minerStake * weightedPool / totalStake
actualReward = fixedPart + weightedPart
```

这意味着：

- 每个有资格拿奖励的出块者都会得到固定部分
- 剩余部分按其在奖励资格集合中的质押权重分配

### 在 Staking 内部的奖励分配

Congress 计算出 `actualReward` 后，Staking 会把它拆成：

- 验证者佣金
- 验证者自质押份额
- 委托人份额

如果某个验证者没有委托人，那么原本属于委托人的那部分会保留给验证者自己。

### 奖励资格

一旦验证者被 jailed，它会立即失去奖励资格，即便 epoch 生效集合还没有轮换。

这样可以防止一个已经被处罚的验证者，只因为还留在当前 epoch 缓存中，就继续享受区块奖励。

### 流动性与支付语义

JPoSA 使用延迟提取本金和容错支付机制。

- 撤销委托后的本金会进入解绑期
- 验证者退出后的自质押也会进入解绑期
- 如果当前无法直接转账，奖励或本金提现会被记录为待领取余额

因此，一次成功的 claim 不一定总是同步完成转账，也可能只是生成 `pendingPayout`。

## 矿工（验证者）角色

在 JPoSA 中，矿工就是验证者。他们负责出块、参与治理，并维护验证者服务质量。

### 成为验证者的流程

创世之后，验证者路径如下：

1. 提案阶段
   - 现有活跃验证者发起新增验证者提案
   - 活跃且未 jailed 的验证者进行投票
2. 注册阶段
   - 一旦过半，候选人获得授权
   - 候选人需要在有效窗口内调用 `registerValidator(...)`
   - 候选人需要提供不低于最低要求的自质押
3. 激活阶段
   - 验证者会立刻进入候选集合
   - 但要到下一个 epoch 才进入实际活跃集合

当前默认阈值：

- 最低验证者自质押：`100000 JU`
- 最多活跃验证者数量：`21`

### 验证者核心职责

验证者需要负责：

- 按 Congress 的轮转规则出块
- 保持节点在线和同步
- 参与治理投票
- 维护有竞争力的质押和委托规模
- 管理佣金率和费地址

### 验证者责任与风险

验证者面临多种运行和经济风险：

1. 漏块
   - 第一个阈值会失去交易费收益资格
   - 第二个阈值会被 jailed 并移出验证者管理路径
2. 双签
   - 会触发 jail 和 slash
3. 延迟退出
   - 不能直接从活跃集合中即时退出
4. 治理依赖
   - 处罚后重新进入需要新的治理提案

在当前 PoSA 机制下，验证者是一个明确的“高收益 / 高风险”角色：

- 验证者可获得 commission
- 验证者可获得 self-stake 对应的奖励份额
- 验证者可通过 feeAddr 获取交易手续费收益
- 直接 slash 风险落在验证者自己的 self-stake 上

### 验证者退出与回归

主动退出是分阶段进行的：

1. `resignValidator()`
2. 等待下一个 epoch 把自己从活跃集合移除
3. `exitValidator()`
4. 再经过解绑期取回本金

从 jail 状态回归也同样分阶段：

1. 先度过 jail 期
2. 重新获得治理授权
3. 保持满足最低自质押要求
4. 调用 `unjailValidator(...)`
5. 等待下一个 epoch 才真正重新活跃

## 用户（委托人）角色

委托人通过经济支持验证者，分享验证者的奖励路径。
在当前 PoSA 设计中，委托人分享收益，但其 delegation principal 与已进入 unbonding 的 principal 不承担直接 slash。

### 委托功能

委托人可以：

- 选择要支持的活跃验证者
- 向一个或多个验证者委托 JU
- 领取属于自己的质押奖励
- 撤销委托并在解绑期后取回本金

当前默认阈值：

- 最低委托：`10 JU`
- 最低撤销委托：`1 JU`

### 解绑与提现

委托本金在发起撤销后不会立刻恢复流动性。

流程是：

1. `undelegate(...)`
2. 本金进入解绑期
3. 到期后调用 `withdrawUnbonded(...)`

奖励提现和本金提现是分开的，奖励不需要走同样的解绑路径，但在直接转账失败时同样可能被记录为待领取余额。

### 委托策略考虑

委托人应当关注：

- 在线率和出块质量
- 治理参与情况
- 佣金率
- 惩罚历史
- 质押深度和长期稳定性

委托不仅是收益选择，也会影响验证者排序和未来活跃集合的构成。
不过，委托人并非完全无风险：即使不承担直接 slash，本质上仍承担验证者表现不佳、被 jailed / remove、收益中断，以及解绑等待时间带来的间接风险。

## 治理委员会角色

治理委员会就是当前 epoch 中活跃且未 jailed 的验证者集合。

### 治理职责

委员会负责：

- 审议并决定验证者的加入与回归
- 在必要时通过治理移除验证者
- 更新经济和运行参数
- 维护验证者集合的质量和网络安全

### 提案类型

委员会投票处理两类提案：

1. 验证者准入和移除提案
2. 参数更新提案

参数更新覆盖：

- 提案时长
- 惩罚阈值
- 奖励输入
- 解绑期和 unjail 期
- 验证者上限和最低门槛
- 佣金上限与冷却
- slash 和举报奖励

### 投票机制

当前投票机制有四个关键特征：

- 只有活跃且未 jailed 的验证者能投票
- 门槛是一半以上
- 一旦过半立即生效
- epoch 区块上禁止投票

这让 JPoSA 的治理既明确高效，又受限于当前活跃验证者集合。

## 可治理参数

当前可治理参数可以按四组理解。

### 时间相关参数

- `proposalLastingPeriod = 604800` blocks
- `withdrawProfitPeriod = 86400` blocks
- `unbondingPeriod = 604800` blocks
- `validatorUnjailPeriod = 86400` blocks
- `doubleSignWindow = 86400` blocks
- `commissionUpdateCooldown = 604800` blocks
- `proposalCooldown = 100` blocks

### 惩罚相关参数

- `punishThreshold = 24`
- `removeThreshold = 48`
- `decreaseRate = 24`
- `doubleSignSlashAmount = 50000 JU`
- `doubleSignRewardAmount = 10000 JU`
- `burnAddress = 0x000000000000000000000000000000000000dEaD`

### 奖励相关参数

- `blockReward = 0.2 JU`
- `baseRewardRatio = 3000` bps
- `maxCommissionRate = 6000` bps

### 验证者和委托参数

- `minValidatorStake = 100000 JU`
- `maxValidators = 21`
- `minDelegation = 10 JU`
- `minUndelegation = 1 JU`

这些参数让网络可以在不重部署系统合约的情况下调整验证者经济和运行安全边界。

## 总结

### 核心价值

JPoSA 围绕三个核心价值构建：

1. 可控的去中心化
   - 质押重要，但准入受治理控制
2. 确定性的运行行为
   - 共识使用父状态选集和 epoch 激活
3. 更强的运行韧性
   - 惩罚、奖励排除和待领取支付都被明确建模

### 设计意义

最终形成的验证者模型同时具备：

- PoA 风格的准入控制
- PoS 风格的委托和权益竞争
- 通过治理暴露的政策调节能力
- 候选排序和 epoch 生效集合分离的运行模型

这正是 JuChain PoSA 的定义性特征。
